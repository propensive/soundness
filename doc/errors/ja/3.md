# `SN-ja/3` — `RecordSchemaError`

Raised when a JSON value does not satisfy the schema it is being
validated against.

## Cause

Jacinta's `records` component validates JSON values against a schema
and reports the first mismatch via this error. The message is `the
JSON was not valid according to the schema because $reason`.

## Variants

### `SN-ja/3.1` — `JsonType`

The value's type does not match what the schema requires.

**Example.** `[↯SN-ja/3.1] the JSON was not valid according to the schema because expected JSON type number, but found string`

**Resolution.** Provide a value of the type the schema requires, or
relax the schema if the type is genuinely flexible.

### `SN-ja/3.2` — `MissingValue`

A required value is absent from the JSON.

**Example.** `[↯SN-ja/3.2] the JSON was not valid according to the schema because the value was missing`

**Resolution.** Add the missing field, or mark it optional in the
schema.

### `SN-ja/3.3` — `IntOutOfRange`

An integer value falls outside the `minimum`/`maximum` allowed by
the schema.

**Example.** `[↯SN-ja/3.3] the JSON was not valid according to the schema because the integer 150 is out of range`

**Resolution.** Constrain the value to the schema's permitted range.

### `SN-ja/3.4` — `PatternMismatch`

A string value does not match the regular expression in the schema's
`pattern`.

**Example.** `[↯SN-ja/3.4] the JSON was not valid according to the schema because the value did not conform to the regular expression [a-z]+`

**Resolution.** Adjust the value to match the schema's pattern, or
loosen the pattern.
